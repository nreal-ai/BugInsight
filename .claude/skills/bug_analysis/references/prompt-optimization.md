# LLM Prompt 优化记录 (2026-04-30)

## 优化目标
根据 qwen3.6-plus 的 32K 上下文窗口，充分利用空间注入高价值信息，同时去除冗余内容。

## 新增注入内容

### 1. Bug 基本信息 (精简版)
```
## Bug 基本信息
- 标题: 【我的眼镜1.7.0】使用手势切换DOF 时我的眼镜崩溃【偶现】
- 描述: 我的眼睛：1.7.0 1983...
- 状态: Aborted
```
- **来源**: `analysis_result["bug_info"]`
- **预算**: 每字段最多 300 字符
- **价值**: 让 LLM 理解缺陷的上下文和现象

### 2. 错误摘要增强 (summary 字段)
```
## 错误摘要
- 错误数: 106
- 警告数: 21
- 致命错误: 50
- 检测标志: crash, native_crash, stack_trace, shader_error, timing_issue
- 错误率: 6.0%
- 分析行数: 1,762
- 日志格式: logcat
```
- **来源**: `log_analysis["summary"]` + `log_analysis["format"]`
- **预算**: 动态生成，约 200 字符
- **价值**: 给 LLM 全局判断依据（哪些类型的检测命中）

### 3. 飞书结论性信息 (新增)
```
## 飞书结论性信息
### 解决方案/最终结论:
- [1742468643000] 问题已解决。NRSDK Version: 3.0.0...
- [1742872383000] 问题已重新修改。NRSDK Version: 3.0.0...
```
- **来源**: `analysis_result["comments"]` 中匹配关键词的评论
- **关键词**: `问题已解决`, `问题已重新修改`, `修复`, `commit`, `Commit`, `修改方案`, `修改内容`, `修改函数`
- **预算**: 最多 3 条，每条 400 字符
- **价值**: 直接给出根因和修复方案

## 已删除的冗余内容

### 1. "飞书已知信息" 中的原始开发者评论
- **原因**: 与"飞书技术证据"大量重复（后者是已清洗的版本）
- **替换**: 仅保留附件列表，评论内容移到"飞书结论性信息"

### 2. 附件列表的冗余换行格式
- **旧格式**: 多行列表，每条带预算检查
- **新格式**: 单行 `日志附件 (4 个): 0217.log, 1459.log, ...`

## 优化原则

1. **高价值优先**: Bug 描述、全局标志位、最终解决方案 > 原始评论
2. **去重**: 避免"飞书已知信息"与"飞书技术证据"重复
3. **动态预算**: 根据模型上下文窗口调整，qwen3.6-plus 32K 窗口充分利用
4. **信息密度**: 每个 section 携带明确的诊断价值，不是单纯的数据堆砌
5. **扩展性**: 若更换模型（如 128K 窗口），可进一步放宽日志预算
